INVITATION TO TENDER

Tender Reference: GOV-2026-IT-0512
Title: Enterprise Data Analytics Platform
Issuing Agency: Department of Finance, Australian Government
Closing Date: 30 April 2026, 5:00 PM AEST

PART A — GENERAL INFORMATION

1. BACKGROUND AND CONTEXT

1.1 The Department of Finance (the "Department") is seeking proposals from qualified vendors to design, develop, deploy, and support an Enterprise Data Analytics Platform (the "Platform"). This Platform will serve as the central analytics capability for the Department, consolidating data from over 50 source systems and providing advanced analytics, machine learning capabilities, and real-time dashboards to approximately 2,000 end users across multiple divisions.

1.2 The Department currently operates a legacy reporting environment based on Oracle Business Intelligence Enterprise Edition (OBIEE) version 11g, which has reached end-of-life. The existing environment processes approximately 3TB of data daily and supports around 500 active report users. The new Platform must significantly expand these capabilities while maintaining backward compatibility with existing reports during a 12-month transition period.

1.3 The Australian Government's Digital Transformation Strategy requires all new ICT investments to be cloud-native, API-first, and designed for continuous delivery. The Platform must align with the Department's Cloud First policy and the Australian Government's Hosting Strategy, which mandates that all data classified at PROTECTED level or below must be hosted in certified Australian data centres.

1.4 This tender is being conducted under the Commonwealth Procurement Rules (CPRs) and the Department's Accountable Authority Instructions. All responses must comply with the requirements set out in this document and any addenda issued prior to the closing date.

2. SCOPE OF WORK

2.1 Data Ingestion and Integration Layer

2.1.1 The Platform must support ingestion of structured data from relational databases including Oracle 19c, PostgreSQL 15+, Microsoft SQL Server 2019+, and MySQL 8.0+. Real-time change data capture (CDC) must be supported for all listed database types with latency not exceeding 30 seconds from source commit to Platform availability.

2.1.2 Semi-structured data ingestion must support JSON, XML, Avro, Parquet, and CSV formats. The Platform must handle schema evolution gracefully, including backward-compatible schema changes without manual intervention and forward-compatible changes with configurable alerting and fallback strategies.

2.1.3 Unstructured data processing must support PDF documents (including scanned documents via OCR), Microsoft Office formats (Word, Excel, PowerPoint), email archives (PST, MBOX), and image files. Natural Language Processing capabilities must be integrated to extract entities, sentiment, and key phrases from unstructured text sources.

2.1.4 API-based ingestion must support REST and GraphQL endpoints with OAuth 2.0 and API key authentication. The Platform must include a self-service API connector framework allowing Department analysts to configure new API data sources without vendor assistance. Rate limiting, retry logic with exponential backoff, and circuit breaker patterns must be implemented.

2.1.5 The ingestion layer must process a minimum of 10 terabytes of raw data per day across all source types combined, with the ability to scale to 50TB per day within the contract period. Peak ingestion rates must support bursts of up to 5x the average daily rate without degradation in downstream query performance.

2.2 Data Storage and Processing

2.2.1 The Platform must implement a modern data lakehouse architecture combining the flexibility of data lake storage with the performance and governance capabilities of a data warehouse. The storage layer must support Delta Lake, Apache Iceberg, or equivalent open table format to ensure data versioning, time travel queries, and ACID transaction guarantees.

2.2.2 Data must be organized in a medallion architecture (Bronze/Silver/Gold layers) with automated data quality checks at each transition. Bronze layer: raw data in original format with ingestion metadata. Silver layer: cleansed, deduplicated, and conformed data with standardized schemas. Gold layer: aggregated, business-ready datasets optimized for specific analytical use cases.

2.2.3 The processing engine must support both batch and stream processing paradigms. Batch processing must handle daily and weekly data transformation jobs completing within a 4-hour processing window for the full 10TB daily volume. Stream processing must support Apache Kafka or equivalent message streaming with exactly-once processing semantics and sub-second latency for real-time analytics use cases.

2.2.4 Data governance capabilities must include automated data cataloguing with business glossary integration, data lineage tracking from source to consumption, automated PII detection and masking, and role-based access control at the column level. The Platform must integrate with the Department's existing Microsoft Entra ID (formerly Azure AD) for identity management.

2.2.5 Data retention policies must be configurable per dataset with support for time-based retention (minimum 7 years for financial data), legal hold capabilities, and automated archival to cold storage tiers. The Platform must support the National Archives of Australia's records management requirements.

2.3 Analytics and Visualization

2.3.1 The Platform must provide a web-based analytics interface supporting self-service report creation, interactive dashboards, and ad-hoc data exploration. The interface must be responsive and function on desktop browsers (Chrome, Edge, Firefox) and tablet devices (iPad, Android tablets) without native app installation.

2.3.2 Dashboard capabilities must include: real-time data refresh with configurable intervals (minimum: 15-second refresh), drill-down and drill-through navigation, cross-filtering between visualizations, dynamic parameter controls, conditional formatting and alerting, export to PDF, Excel, and PowerPoint formats, and scheduled distribution via email with role-based content filtering.

2.3.3 The Platform must include a natural language query interface allowing users to ask questions about their data in plain English and receive visualized responses. The NLQ engine must understand the Department's business terminology (configurable via the business glossary) and support follow-up questions that maintain conversational context.

2.3.4 Advanced analytics capabilities must include: statistical analysis (regression, correlation, hypothesis testing), time series forecasting with multiple algorithm support (ARIMA, Prophet, neural network-based), clustering and segmentation analysis, anomaly detection with configurable sensitivity, and what-if scenario modeling. These capabilities must be accessible to business analysts through a point-and-click interface without requiring programming knowledge.

2.3.5 Geospatial analytics must support Australian geographic data standards including the Australian Statistical Geography Standard (ASGS). The Platform must render choropleth maps, point maps, heat maps, and flow maps using Australian boundary datasets. Integration with the Geocoded National Address File (G-NAF) is required for address-level analytics.

2.4 Machine Learning and AI

2.4.1 The Platform must include a managed ML model development environment supporting Python 3.10+ and R 4.0+ with integrated Jupyter notebooks or equivalent. The environment must provide GPU-accelerated computing for model training with support for NVIDIA A100 or equivalent GPUs.

2.4.2 An MLOps pipeline must be provided including: experiment tracking and model versioning, automated model training and hyperparameter tuning, model registry with approval workflows, A/B testing and canary deployment capabilities, model performance monitoring with automated drift detection, and automated model retraining triggers based on performance degradation thresholds.

2.4.3 Pre-built AI models must be available for common government analytics use cases including: document classification and routing, named entity recognition for Australian government entities, sentiment analysis calibrated for Australian English, predictive resource demand forecasting, and fraud detection patterns. These models must be fine-tunable using Department-specific training data.

2.4.4 The Platform must support responsible AI practices including: model explainability reports (SHAP values, feature importance), bias detection and mitigation tools, model cards documenting intended use, limitations, and ethical considerations, and audit trails for all model predictions used in decision-making processes.

2.5 Security and Compliance

2.5.1 The Platform must be assessed and certified under the Information Security Registered Assessors Program (IRAP) at the PROTECTED level in accordance with the Australian Government Information Security Manual (ISM). The vendor must hold a current IRAP assessment or commit to completing assessment within 6 months of contract execution.

2.5.2 All data must be encrypted at rest using AES-256 or equivalent and in transit using TLS 1.3. Key management must use a FIPS 140-2 Level 3 certified Hardware Security Module (HSM) or equivalent. The vendor must support the Department's Bring Your Own Key (BYOK) requirements.

2.5.3 Access control must implement Zero Trust Architecture principles with: multi-factor authentication for all users, session management with configurable timeout and concurrent session limits, privileged access management for administrative functions, and comprehensive audit logging of all data access and system changes. Audit logs must be tamper-evident and retained for a minimum of 7 years.

2.5.4 The Platform must comply with the Privacy Act 1988 (Cth), Australian Privacy Principles, the Notifiable Data Breaches scheme, and the Security of Critical Infrastructure Act 2018 (SOCI Act). The vendor must demonstrate compliance with the Essential Eight maturity model at Maturity Level 3 or above.

2.5.5 Vulnerability management must include: automated vulnerability scanning at least weekly, penetration testing at least annually by an independent assessor, a documented vulnerability disclosure and remediation process with SLA (Critical: 24 hours, High: 72 hours, Medium: 30 days, Low: 90 days), and participation in the Australian Cyber Security Centre's (ACSC) threat intelligence sharing program.

PART B — COMMERCIAL REQUIREMENTS

3. MANDATORY REQUIREMENTS

Proposals that do not meet ALL mandatory requirements will be excluded from further evaluation.

3.1 The vendor must hold current ISO 27001:2022 certification covering the services proposed.

3.2 The vendor must demonstrate a minimum of 3 years experience delivering data analytics platforms for Australian Government agencies at the Commonwealth or State/Territory level, evidenced by at least 2 reference sites of comparable scale (processing 5TB+ daily data volume).

3.3 The vendor must hold a current IRAP assessment at PROTECTED level, or provide a binding commitment to achieve IRAP PROTECTED assessment within 6 months of contract execution, with liquidated damages of AUD 50,000 per month of delay.

3.4 The vendor must demonstrate proven expertise in Kubernetes and cloud-native deployment architectures, evidenced by certified Kubernetes administrators (minimum 3 CKA-certified staff) on the proposed project team.

3.5 All proposed hosting infrastructure must be located within Australia in data centres certified to at least Uptime Institute Tier III or equivalent. The vendor must name the specific data centres to be used.

3.6 The vendor must maintain the following insurance coverage throughout the contract term: Professional Indemnity Insurance of not less than AUD 20 million per claim, Public Liability Insurance of not less than AUD 20 million per occurrence, and Cyber Liability Insurance of not less than AUD 10 million per claim.

5. BUDGET AND PRICING

5.1 The estimated total contract value is AUD 8 million to AUD 12 million over the initial 3-year term, comprising implementation costs and ongoing operational costs.

5.2 Pricing must be provided in the following categories:
- Implementation and deployment (fixed price)
- Annual platform licensing (per-user and/or capacity-based)
- Managed services and support (annual)
- Training and change management (fixed price)
- Optional: additional consulting services (daily rate card)

5.3 All prices must be in Australian Dollars, GST-exclusive, and held firm for the initial 3-year term. Price adjustments for extension periods must not exceed CPI + 2% per annum.

5.4 Payment will be milestone-based for implementation, with monthly invoicing for ongoing services. Payment terms: 30 days from receipt of correctly rendered tax invoice following milestone acceptance.

7. PENALTIES AND SERVICE LEVELS

7.1 Late Delivery Liquidated Damages
- Milestone delays: AUD 10,000 per business day per milestone, capped at 15% of the milestone payment value.
- Go-Live delay: AUD 25,000 per business day, capped at 10% of total implementation cost.
- These liquidated damages represent a genuine pre-estimate of the Department's loss.

7.2 Operational Service Levels
- Platform Availability: 99.95% measured monthly (excluding planned maintenance windows of up to 4 hours per month during off-peak hours)
- Query Response Time: 95th percentile of dashboard load times must not exceed 5 seconds for standard reports, 15 seconds for complex analytical queries
- Data Freshness: Real-time data sources must reflect in the Platform within 60 seconds; batch sources within 4 hours of source system batch completion
- Incident Response: P1 (Critical — platform unavailable): 15 minutes acknowledgment, 1 hour resolution target; P2 (High — major feature impaired): 30 minutes acknowledgment, 4 hours resolution target; P3 (Medium — minor impact): 2 hours acknowledgment, 1 business day resolution target

7.3 Service Credit Regime
- Availability below 99.95%: 3% of monthly service fee rebate per 0.01% below target
- Availability below 99.5%: Termination right triggered
- Maximum monthly service credits: 25% of monthly service fee
- Service credits are the Department's sole and exclusive remedy for SLA failures, except where termination rights are triggered

7.4 Cumulative Liquidated Damages Cap
- Total liquidated damages under this contract shall not exceed 10% of the total contract value
- If the cumulative LD cap is reached, the Department may terminate the contract for cause

8. CONTRACT TERMS

8.1 The initial contract term is 3 years from the date of contract execution, with options for two 2-year extensions exercisable at the Department's sole discretion.

8.2 The Department may terminate the contract: for convenience with 90 days written notice (payment for work completed plus reasonable wind-down costs); for cause with 30 days written notice and cure period for material breach; immediately if the vendor enters insolvency or fails to maintain mandatory certifications.

8.3 Upon termination or expiry, the vendor must: provide transition assistance for up to 6 months at rates not exceeding the contracted daily rates; export all Department data in open, documented formats; destroy all copies of Department data within 30 days of transition completion and provide certification of destruction; transfer all Department-specific configurations, customizations, and documentation.

8.4 Intellectual Property: All IP in pre-existing vendor software remains with the vendor. All IP in Department data remains with the Department. IP in customizations and configurations developed specifically for the Department shall vest in the Department upon payment.

8.5 The contract is governed by the laws of the Australian Capital Territory. Disputes shall be resolved through negotiation, then mediation under the ACDC Mediation Guidelines, then arbitration under the ACICA Arbitration Rules.
